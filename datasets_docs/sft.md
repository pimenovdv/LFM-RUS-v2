# Контролируемое дообучение (General SFT)

## Формат данных
На этапе SFT (Supervised Fine-Tuning) модель обучается следовать инструкциям и вести диалог. Датасеты должны быть в диалоговом формате, таком как **ChatML** или стандартном формате сообщений Hugging Face.

Формат JSONL, где каждая запись представляет собой массив сообщений:
```json
{"messages": [{"role": "system", "content": "You are a helpful assistant."}, {"role": "user", "content": "Объясни квантовую механику."}, {"role": "assistant", "content": "Квантовая механика — это раздел физики..."}]}
```
SFTTrainer из `trl` поддерживает функцию упаковки последовательностей (sequence packing), объединяя несколько коротких диалогов в единый блок, равный максимальному контекстному окну, разделяя их EOS-токенами.
Смесь данных (datamix) для SFT должна быть широкой, покрывая ролевые игры, экстракцию фактов, переписывание, код, математику, чтобы трансформировать базовую модель в универсального ассистента.

## Требуемое количество данных (Количество диалогов/примеров)
Количество примеров для SFT масштабируется вместе с размером модели, чтобы модель могла выучить разнообразные паттерны поведения, не переобучаясь (overfitting) на малом наборе. Качество данных здесь важнее их количества (Quality > Quantity).

Ориентировочные объемы для General SFT (предполагая среднюю длину диалога ~500-1000 токенов):

* **100m**: **~10,000 - 25,000** высококачественных диалогов.
* **350m**: **~25,000 - 50,000** высококачественных диалогов.
* **0.5b**: **~40,000 - 80,000** высококачественных диалогов.
* **1b**: **~80,000 - 150,000** диалогов.
* **2b**: **~150,000 - 250,000** диалогов.
* **4b**: **~250,000 - 400,000** диалогов.
* **8b**: **~400,000 - 800,000** диалогов (до 1 млн).

*(Использование большего количества данных возможно, но при этом критически важно соблюдать баланс классов и следить за кривой валидации, чтобы избежать деградации разнообразия генерации).*
