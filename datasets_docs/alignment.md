# Выравнивание (Alignment - DPO / GRPO)

## Формат данных
На этапе Alignment модель выравнивается с человеческими предпочтениями. Формат данных зависит от выбранного алгоритма:

### Для DPO (Direct Preference Optimization):
Требуется формат преференций: промпт (prompt), выбранный ответ (chosen) и отвергнутый ответ (rejected).
```json
{
  "prompt": "Как отсортировать список в Python?",
  "chosen": "В Python вы можете использовать метод `.sort()` или функцию `sorted()`. Вот пример...",
  "rejected": "Используйте цикл for."
}
```

### Для GRPO (Generalized Reward Policy Optimization):
Требуется формат с промптами и возможностями для вычисления функции вознаграждения (reward functions), например, для задач рассуждения или математики.
```json
{
  "prompt": "Реши уравнение 2x + 4 = 10",
  "expected_answer": "3"
}
```
*(GRPO будет генерировать несколько ответов и оценивать их на основе `expected_answer` или правил оформления, например, наличия тега `<answer>`).*

## Требуемое количество данных
Для этапов Preference Tuning (DPO/ORPO/GRPO) требуется меньше данных, чем для SFT, так как модель уже умеет следовать инструкциям; задача состоит в тонкой калибровке вероятностей генерации желаемых/нежелательных токенов.

* **100m**: **~5,000 - 10,000** пар (троек) преференций.
* **350m**: **~10,000 - 20,000** пар.
* **0.5b**: **~15,000 - 30,000** пар.
* **1b**: **~25,000 - 50,000** пар.
* **2b**: **~40,000 - 80,000** пар.
* **4b**: **~60,000 - 120,000** пар.
* **8b**: **~100,000 - 200,000** пар.

*(Для алгоритма GRPO (RLVR) количество промптов может быть меньше, так как одна задача используется для генерации множества сэмплов (rollouts), но требуется высокая вычислительная мощность на генерацию).*
